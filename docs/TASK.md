## 1. Вступ

Це технічне завдання поширюється на розробку комп'ютерної гри у жанрі «виживання + симулятор ферми» з робочою назвою **«Зелений Рубіж: Кисневий Порятунок»**. Продукт є інтерактивною системою з перспективою «зверху-вниз» (top-down ортогональна проєкція), що поєднує механіки управління ресурсами, вирощування культур та поступового відновлення екосистеми мертвої планети. 

Актуальність розробки зумовлена високою популярністю жанру та необхідністю практичного застосування навичок об'єктно-орієнтованого програмування, архітектурних патернів та кросплатформної розробки цифрових рішень.

* **Тема розробки:** *«Створення 2D-гри жанру симулятор ферми з елементами виживання на базі ООП»*.

## 3. Призначення розробки

Програмний продукт призначений для розважальних цілей, забезпечення інтерактивного ігрового досвіду для користувачів та демонстрації можливостей фреймворку LibGDX при побудові комплексних стейт-машин і систем серіалізації даних.

**Основний ігровий сценарій:**
Головний герой опиняється на космічній планеті, позбавленій кисню. Єдиним живим об'єктом на всій території є деградуюче **Серцевинне Дерево**. Гравець повинен забезпечувати його життєдіяльність, розробляти ферму, вирощувати спеціальні рослини для генерації О₂, досліджувати навколишню територію та здійснювати торговельні операції із Землею. 

**Кінцева мета гри** — повністю відновити атмосферу планети, перевівши Дерево у фінальну фазу еволюції, та зробити локацію придатною до життя.

## 4. Вимоги до програми та програмного виробу

### 4.1. Вимоги до функціональних характеристик

Програма повинна забезпечувати стабільне виконання наступних геймплейних та системних функцій:

* **Управління (Mouse Input):** Здійснюється виключно мишею. 
    * `Click` — взаємодія з об'єктами інтерфейсу, вибір тайлів, посадка та збір врожаю, переміщення персонажа по клітинках сітки.
    * `Drag` — динамічне переміщення камери по карті, прокручування списків в інвентарі чи магазині.
* **Механіка кисню (Oxygen Manager):** Наявність показника місткості балона скафандра. Кисень безперервно витрачається під час переміщення персонажа по планеті поза куполом. Поповнення балона та безпечне перебування можливе лише в межах зони дії ферми або Серцевинного Дерева.
* **Top-down фермерство (Farming System):** Можливість взаємодії з прямокутною 2D-сіткою тайлів (`Grid`) для створення грядок, посадки насіння, поливу та збору дозрілого врожаю. Рослини виступають одночасно джерелом генерації кисню та економічним ресурсом.
* **Розвиток Серцевинного Дерева (Evolution System):** Система з 5 послідовних фаз розвитку. Настання кожної нової фази супроводжується візуальною трансформацією карти (тайли довкола стають зеленими, змінюється колір загального освітлення) та геометричним розширенням безпечної кисневої зони.
* **Торгівля (Rocket Terminal):** Наявність ракетного терміналу для зв'язку із Землею. Забезпечує обмін зібраного врожаю на внутрішню валюту ($), купівлю нових типів насіння, технічних апгрейдів (покращення скафандра, розширення резервуарів для води) чи будівельних пілонів.
* **Диференціація рівнів складності:**
    * `Легкий (Easy)`: Повільна деградація кисню в балоні, низькі ціни на апгрейди, відсутність жорстких штрафів за посуху.
    * `Нормальний (Normal)`: Стандартний виважений баланс між швидкістю виживання та швидкістю накопичення капіталу.
    * `Складний (Hard)`: Прискорена витрата О₂, високі ціни в терміналі, випадкові негативні екологічні події (наприклад, раптова посуха грядок), видалення файлу збереження після смерті гравця (Permadeath).

### 4.2. Вимоги до надійності
* **Обробка виключних ситуацій (Exception Handling):** Система повинна коректно валідувати дії користувача на рівні бізнес-логіки. Спроби придбати товар за відсутності грошей, посадити насіння за відсутності його в інвентарі або зібрати невизрілу культуру повинні блокуватися без виклику критичних помилок (на кшталт `NullPointerException`).
* **Серіалізація стану (Save/Load):** Програма повинна мати архітектурний модуль автоматичного або ручного збереження повного стану ігрового світу: поточних координат гравця, рівня кисню, фази розвитку Дерева, масиву грядок з їхніми таймерами, а також `HashMap` інвентарю.

### 4.3. Вимоги до інформаційної та програмної сумісності
* **Мова програмування:** Java 17 (з використанням сучасних конструкцій, `records`, `sealed` класів та покращених `switch`).
* **Ігровий рушій / Фреймворк:** **LibGDX**. Використовується як низькорівневе ядро для рендеру графіки (OpenGL контекс), менеджменту екранів (`Scene2D`), обробки потоків введення з миші (`InputProcessor`), відтворення аудіо та базових векторних обчислень.
* **Менеджмент залежностей:** Автоматизована система збірки **Maven** (файл `pom.xml` для керування сторонніми бібліотеками).
* **Модульне тестування:** Покриття чистої ігрової логіки, математичних розрахунків `OxygenManager`, CRUD-операцій інвентарю та логіки таймерів росту за допомогою фреймворку **JUnit 5**.
* **Формати даних:**
    * `JSON (Google GSON)` — для збереження/завантаження прогресу гравця.
    * `.tmx (Tiled Map Editor)` — для зберігання структури ортогональних шарів карти, яка зчитується за допомогою `OrthogonalTiledMapRenderer` вбудованого в LibGDX.
* **Цільова платформа:** Операційні системи сімейства Windows (10/11). Продукт постачається у вигляді десктопного застосунку із точкою входу в класі `DesktopLauncher`.
